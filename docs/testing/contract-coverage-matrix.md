# Contract Coverage Matrix

Status: draft

This matrix maps every active contract ID to current evidence. To keep the file
maintainable, rows may group IDs from the same contract file when they share the
same owner module and evidence tier. The contract files keep finer per-ID notes.

Allowed status values:

| Status | Meaning |
| --- | --- |
| `covered` | Direct evidence exists and is appropriate for the row priority. |
| `partial` | Some behavior is covered, but important variants or failures are missing. |
| `missing` | No direct evidence exists yet. |
| `deferred` | Evidence belongs in benchmark, replay, nightly, or future feature work. |

Priority levels:

| Priority | Meaning |
| --- | --- |
| `P0` | Must be covered before related behavior is changed or pushed. |
| `P1` | Should be covered by focused follow-up evidence; replay or integration evidence may supplement. |
| `P2` | Benchmark/nightly/documentation tracking is acceptable. |

Evidence tier convention:

- For `P0` and `P1` `covered` rows, the notes should identify the focused or
  specialized test/inspection evidence that closes the row. Integration tests,
  replay cases, full `ctest`, and benchmark reports can supplement the row, but
  should not be the only closure evidence unless the contract itself is a
  compile/build/selection/static-policy contract.
- `P2` rows may remain `partial` or `deferred` with an explicit deferred path.

## Remaining Closure Queue

All current `P0` and `P1` rows are covered. Remaining `partial` or `deferred`
rows are `P2` and belong to optional direction-refinement replay, logging or
build CI variants, backend benchmark metadata, and numerical-Jacobian fallback
accuracy evidence.

Do not downgrade a `P1/P2` row to `P0` unless a behavior change touches that
contract or release scope makes the row blocking.
