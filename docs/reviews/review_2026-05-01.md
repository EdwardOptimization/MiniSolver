## Overall Architecture

### Strengths

- Module boundaries are clear: integrator, matrix, solver, and generator stay separated.
- SFINAE dispatch is clean: `has_fused_riccati_step`, `has_generated_integrator`, and `has_dynamics_continuous` provide compile-time routing with no runtime cost.
- The Eigen and MiniMatrix backends share the `MatOps` interface through `matrix_defs.h`.
- The generator sparsity-pattern analysis uses graph reachability, including Floyd-Warshall transitive closure where needed.
- Failure propagation moved from silent fallbacks such as `A=I, B=0` to explicit NaN marking or exceptions.
- `evaluate_model_stage` removed repeated cost, dynamics, and constraint evaluation call sites.
- Both backends passed 19/19 CTest entries. MiniMatrix had already gained `head`, `tail`, `block`, `topRows`, and `bottomRows`.

### Risks

- `has_generated_integrator` existed in both `solver.h:detail` and `riccati.h:internal`. Both copies were functionally correct, but this was a maintenance risk.
- Implicit `integrate()` and `compute_dynamics()` each implemented Newton logic. The semantics are different, because `integrate()` only rolls out `x_next`, while `compute_dynamics()` also produces `f_resid`, `A`, and `B`. A shared helper would need a deliberate `integrate_and_linearize` style interface.

## Module 1: Implicit Integrator

### Strengths

- Backward Euler, Implicit Midpoint, and 2-stage Gauss-Legendre were implemented.
- The Gauss-Legendre Butcher tableau coefficients were verified.
- Discrete Jacobians were derived through the implicit function theorem.
- `dispatch_compute_dynamics` and `dispatch_integrate` provided a unified entry point and were used by line search.
- Newton non-convergence returned invalid data explicitly instead of silently setting `A=I, B=0` (`5262f5f`).
- Gauss-Legendre reused a single LU factorization and solved the combined `[Jx; Ju]` right-hand side (`e4eaf24`).
- The Newton evaluation lambda preserved the final Jacobian for reuse after convergence (`e4eaf24`).

### Findings

| Priority | Finding | Status | Notes |
|----------|---------|--------|-------|
| Medium | Numerical Jacobian cost inside Newton | Open, limited impact | Only affects handwritten models without `jacobian_continuous()`. Python generated models provide analytic continuous Jacobians through SFINAE detection. |
| Low | Duplicate Newton logic in `integrate()` and `compute_dynamics()` | Deferred by design | Calls were unified through `evaluate_model_stage`, but the lower-level helper needs an explicit API design first. |

## Module 2: Newton Solver

### Strengths

- Zero heap allocation; all work buffers are static or stack allocated.
- The public call shape is compact: `solve(x, eval, config)`.
- The solver reevaluates `eval_func(x, F_, J_)` after the loop, so convergence checks use the latest residual.

### Findings

| Priority | Finding | Status | Notes |
|----------|---------|--------|-------|
| Medium | Fixed regularization value `1e-12` | Needs stress test first | Adaptive damping should not be added without a failing ill-conditioned Newton test. |
| Low | `warm_start` was effectively dead code | Open | `NewtonSolver` instances were local and `converged_last_` started false, so warm start state was not reused. |

## Module 3: Numerical Jacobian

### Strengths

- Uses centered differences with second-order truncation error.
- Step size is adaptive: `h = eps * max(1.0, abs(value))`.
- Zero heap allocation.

### Reclassified Non-Issues

- Fixed step size was not an issue because the implementation already scaled the step by variable magnitude.
- `eps` type consistency was not an issue because `eps` and `h` are both `double`.

## Module 4: MiniMatrix And Kernel Layer

### Strengths

- `MatrixPolicy` provides compile-time unroll decisions with macro overrides.
- `StaticFor`, `ForRange`, `PrefixRange`, and `SuffixRange` keep kernel loops structured.
- Matrix kernels are zero-malloc and thread-safe.
- Bit-level NaN/Inf checks are safe under `-ffast-math`.
- `MiniLLT` and `MiniLDLT` check `success` in `solve_in_place`; failure returns zero (`c98669e`).
- Block reference constructors include boundary `assert`s (`c98669e`).
- Static view helpers include compile-time boundary checks.
- LU solve uses `std::array`, not a variable-length array.
- `dot()` casts both operands to `double` before multiplication.

### Findings

| Priority | Finding | Status | Notes |
|----------|---------|--------|-------|
| Low | Branch cost in unrolled `PrefixRange` and `SuffixRange` | Open | Generates `if (i < count)` guards under unrolling. Impact is expected to be small for current small LDLT loops. |
| Low | `SymmetrizeImpl<true>` visits all `N*N` elements | Open | Non-unrolled implementation only visits the upper triangle. |
| Low | `has_nan`, `all_finite`, and `norm_inf` do not use the policy system | Open | Mostly a consistency concern because the functions are simple. |

## Module 5: Riccati Solver

### Strengths

- Backward pass, Schur complement, forward sweep, and dual recovery were mathematically consistent.
- `fast_inverse` had specialized 1x1, 2x2, and 3x3 paths with SPD checks.
- SPD fallback covered regularization, singular ignore, and saturation strategies.
- Hard constraints, L1 soft constraints, and L2 soft constraints had dedicated barrier derivative and dual recovery logic.
- Defect correction used the sign convention `defect = f_resid - x_next`.
- `dsoft_s` was included in NaN checks (`9390056`).
- L1 sigma used `positive_barrier_gap()` to clamp `w - lam` and avoid division by zero (`9390056`).
- The fused-kernel contract was documented (`9390056`).

### Findings

| Priority | Finding | Status | Notes |
|----------|---------|--------|-------|
| Low | Default `fast_inverse` epsilon `1e-9` may be too strict | Open | Callers passing `config.singular_threshold` are correct; the default threshold may reject legal but ill-conditioned SPD matrices. |

- Each implicit integrator had a pattern calculation reflecting its mathematical structure.

- `generated_integrator` was emitted correctly.
- Runtime dispatch correctly guarded generated explicit/implicit integrator compatibility.

## Integration Findings

### Duplicate Generated-Integrator Trait

- `solver.h:detail` and `riccati.h:internal` each defined a generated-integrator trait.
- The behavior was correct because the definitions lived in different namespaces.
- A shared trait header would reduce maintenance risk.

### Silent Fallbacks

Status: fixed in the reviewed range.

- `implicit_integrator.h`: `A=I, B=0` changed to `mark_jacobians_invalid(kp)` with NaN output (`5262f5f`).
- `MiniLLT::solve_in_place` gained a success check (`c98669e`).
- `MiniLDLT::solve_in_place` gained a success check (`c98669e`).
- Unsupported integrator paths now throw instead of silently falling through (`5262f5f`).

### Unified Model Evaluation

Status: fixed in `5cd5945`.

- Added `model_evaluation.h`.
- Six repeated line-search call sites became one helper call.
- Two repeated solver call sites became one helper call.
- The helper uses `dispatch_compute_dynamics`, not `Model::compute_dynamics`, so implicit integrator routing remains correct.
